# ADR-0003: BPMN vs C4 — Выбор инструмента моделирования

**Статус:** `accepted`  
**Дата:** 2026-06-26  
**Автор:** Михаил Прасолов

---

## Контекст

Команда системных аналитиков выбирает инструмент и нотацию для визуального моделирования. Основные задачи:

| Задача | Частота |
|--------|---------|
| Описание бизнес-процессов | Ежедневно |
| Архитектурные схемы систем | Еженедельно |
| Sequence-диаграммы (сценарии) | Ежедневно |
| Интеграционные схемы | Еженедельно |

Встал вопрос: **одна нотация на все случаи или комбинировать?**

---

## Драйверы решения

- **Требование:** Единый стандарт в команде
- **Требование:** Понятность для бизнес-пользователей
- **Требование:** Возможность генерации из текста (DSL/PlantUML)
- **Ограничение:** Инструменты должны быть бесплатными или уже купленными

---

## Рассмотренные варианты

### Вариант 1: BPMN + C4 + PlantUML (выбран)

Каждая нотация для своей задачи:

| Нотация | Для чего | Инструмент |
|---------|----------|------------|
| **BPMN** | Бизнес-процессы, workflow, оркестрация | Camunda Modeler (бесплатно) |
| **C4 (Structurizr DSL)** | Архитектура систем: Context → Container → Component | Structurizr DSL |
| **PlantUML** | Sequence-диаграммы — детализация сценариев | PlantUML (встроен в IDE) |

**Pros:**
- ✅ Каждая нотация решает свою задачу оптимально
- ✅ BPMN понимают бизнес-аналитики и заказчики
- ✅ C4 даёт ровно 4 уровня — не больше, не меньше
- ✅ PlantUML — текстовый, хранится в git, diff читаемый

**Cons:**
- ❌ Три инструмента вместо одного
- ❌ Нужно знать все три нотации
- ❌ Иногда дублирование (процесс в BPMN и последовательность в PlantUML)

### Вариант 2: Только BPMN

**Pros:**
- ✅ Единый стандарт
- ✅ Мощный — можно моделировать практически всё

**Cons:**
- ❌ Сложные архитектурные схемы (C4 Context) выглядят громоздко
- ❌ Неудобен для быстрых sequence-диаграмм
- ❌ Хранение в git — бинарный формат .bpmn, diff нечитаемый

### Вариант 3: Только PlantUML

**Pros:**
- ✅ Текстовый, идеальный git diff
- ✅ Есть разные виды диаграмм (sequence, activity, component)

**Cons:**
- ❌ PlantUML для бизнес-процессов неудобен — нет BPMN-элементов (gateway, pool)
- ❌ C4-диаграммы лучше выглядят в Structurizr
- ❌ Нет нотации, понятной бизнесу

### Вариант 4: Draw.io / Diagrams.net

**Pros:**
- ✅ Бесплатный, простой
- ✅ Любые диаграммы

**Cons:**
- ❌ XML-формат, плохой git diff
- ❌ Нет стандарта — каждый рисует как хочет
- ❌ Трудно автоматизировать генерацию

---

## Решение

**Выбрано:** BPMN + C4 (Structurizr) + PlantUML.

| Задача | Нотация | Инструмент | Формат |
|--------|---------|------------|--------|
| Бизнес-процесс | BPMN 2.0 | Camunda Modeler | `.bpmn` (XML) |
| Архитектура системы | C4 (1-4 уровня) | Structurizr DSL | `.dsl` (текст) |
| Детальный сценарий | Sequence | PlantUML | `.wsd` / `.puml` (текст) |

**Правила:**
- Не дублировать одно и то же в разных нотациях
- Если процесс описан в BPMN — не нужно рисовать sequence на каждый шаг
- Sequence — только для сценариев с внешними системами или сложной логикой
- C4 — только на уровне Container и Component для ключевых сервисов

---

## Последствия

### Позитивные
- ✅ Оптимальный инструмент для каждой задачи
- ✅ BPMN — contracts с бизнесом, C4 — архитектурная документация, PlantUML — dev-детали
- ✅ Все три инструмента бесплатные
- ✅ PlantUML и Structurizr — текст, хранятся в git

### Негативные
- ❌ Выше порог входа — нужно знать 3 нотации
- ❌ Нужно следить, чтобы схемы не расходились с реальностью (CI-проверки)
- ❌ Нет единой "source of truth" диаграммы

### Нейтральные
- ➡️ Можно автоматизировать генерацию BPMN → PlantUML через XSLT (при необходимости)
