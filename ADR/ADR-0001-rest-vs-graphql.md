# ADR-0001: REST vs GraphQL для публичных API

**Статус:** `accepted`  
**Дата:** 2026-06-26  
**Автор:** Михаил Прасолов

---

## Контекст

Проектируем API для интернет-магазина. Основные потребители:
- **Frontend** (веб + мобильное приложение) — нужны кастомизированные запросы с разными наборами полей
- **Партнёры** (внешние системы) — нужны чёткие контракты, предсказуемость
- **Админка** — CRUD-операции над сущностями

Встал вопрос: какой стиль API выбрать для публичных эндпоинтов?

---

## Драйверы решения

- **Требование:** Единый API для frontend и партнёров
- **Требование:** Версионирование и обратная совместимость
- **Ограничение:** Внешние партнёры не могут часто обновлять клиенты
- **Допущение:** Команда знает REST, GraphQL — новый опыт

---

## Рассмотренные варианты

### Вариант 1: REST (выбран)

**Pros:**
- ✅ Стандарт индустрии — все партнёры умеют
- ✅ Простое кэширование (HTTP-кэш, CDN)
- ✅ Инструментарий — OpenAPI, Swagger, Postman — зрелый
- ✅ Чёткие контракты — OpenAPI-спецификация = источник истины
- ✅ Обратная совместимость через версионирование `/v1/`, `/v2/`

**Cons:**
- ❌ Over-fetching / under-fetching — фронтенду приходится делать несколько запросов
- ❌ Множество эндпоинтов — сложнее поддерживать
- ❌ Сложная пагинация и фильтрация для сложных запросов

### Вариант 2: GraphQL

**Pros:**
- ✅ Клиент сам выбирает поля — нет over-fetching
- ✅ Одна точка входа — `/graphql`
- ✅ Сильная типизация — schema-first подход
- ✅ Хорош для rapid prototyping

**Cons:**
- ❌ Сложное кэширование (нет HTTP-методов)
- ❌ Партнёрам сложнее — нужен GraphQL-клиент
- ❌ N+1 проблема — легко написать неэффективный запрос
- ❌ Rate-limiting сложнее реализовать
- ❌ Версионирование — обычно через deprecate полей, неявное

### Вариант 3: REST + BFF (Backend For Frontend)

**Pros:**
- ✅ Промежуточный слой для фронтенда решает over-fetching
- ✅ Партнёрский API остаётся REST
- ✅ Каждый BFF оптимизирован под свой клиент

**Cons:**
- ❌ Дополнительный сервис — больше инфраструктуры
- ❌ Дублирование логики в BFF
- ❌ BFF нужно отдельно поддерживать

---

## Решение

**Выбрано:** REST как основной API-стиль.

Обоснование:
- Партнёры — ключевые потребители API. REST минимальный барьер входа
- OpenAPI-спецификация = контракт, который можно генерировать, тестировать, документировать
- Для frontend используем BFF (отдельное приложение) — он решает проблему over-fetching без внедрения GraphQL
- В будущем можно добавить GraphQL-точку для internal-потребителей, если REST+BFF перестанет покрывать потребности

---

## Последствия

### Позитивные
- ✅ Единый стандарт для всех API в системе
- ✅ Партнёры подключаются по знакомой технологии
- ✅ OpenAPI-спецификации версионируются и документируются
- ✅ Инструменты (Swagger, Redoc, Postman) работают из коробки

### Негативные
- ❌ BFF-слой нужно проектировать и поддерживать отдельно
- ❌ Для сложных запросов с большим количеством связей REST неэффективен

### Нейтральные
- ➡️ При росте числа клиентов BFF может потребоваться больше ресурсов
